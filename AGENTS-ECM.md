## 本地演示产品实现规范

### 组件库优先级
- 用户提到“组件库”时，默认指 Figma `Maycur Design PC beta 1.0`，不是任意本地 CSS。
- 涉及弹窗、按钮、表格、筛选、图标、抽屉等组件尺寸时，优先通过 Figma MCP 读取组件库实际节点尺寸，不要凭截图估算。
- 如果 Figma 暂时访问失败，应先重试并说明访问状态；只有在无法访问且用户同意时，才可参考截图近似还原。

### 设计稿与需求优先级
- Figma 设计稿是视觉还原基准，但如果用户在对话中明确修正了交互逻辑，以用户最新口径为准。
- 当实现结果刻意偏离 Figma 时，必须在交付说明中写清楚偏离原因，例如“用户最新要求覆盖设计稿旧备注”。
- 复查 Figma 时，应区分视觉规范、交互说明、旧批注和当前需求，不要机械照搬过期备注。

### 页面复用规则
- 新增页面应优先复用已有相近页面结构。例如列表页优先复用已有筛选区、表格、分页器、底部操作栏。
- 不要为单个需求另起一套列表、筛选、分页、弹窗样式，除非现有组件无法承载。
- 修改已有页面时，只改当前业务相关区域，避免影响公共组件和其他页面。

### 列表页和分页器
- 所有带分页器的页面，内容区域应保持固定高度，分页器固定在底部，不要因数据量、hover 或文本换行导致页面高度跳动。
- 列表卡片、表格行、时间字段、状态字段应设置稳定宽度和高度约束，hover 时不能让文本、背景块或时间字段变形。
- 长文本允许截断或按设计稿限定行数展示，不要让单条数据破坏整体布局。

### 交互确认规则
- 如果用户明确要求“先告诉我准备改哪些区域，确认后再动代码”，必须先说明修改范围，等用户确认后再写代码。
- 对弹窗、流程跳转、按钮状态这类交互，必须说明会影响哪些入口、路由和关闭逻辑。
- 不确定业务含义时，先给判断和假设，不擅自扩展字段或状态。

### 弹窗规范
- 弹窗尺寸、按钮尺寸、标题字号、图标尺寸以 Figma 组件库实测值为准。
- 弹窗遮罩如要求覆盖整个页面，应覆盖 topbar、sidebar、内容区，不能只覆盖工作区。
- 弹窗应保持水平和垂直居中。
- 取消按钮默认使用普通描边按钮，不要用禁用态模拟；hover 时描边和文字应高亮。

### 图标规范
- 菜单图标应尽量使用 inline SVG symbol，并使用 `currentColor`，确保 hover/active 状态能跟随菜单颜色。
- 替换单个菜单图标时，不要覆盖已有通用 symbol；应新增独立 symbol，避免影响其他业务位置。
- 外部 SVG 资源如需纳入演示页，优先转为内联 symbol，保持单文件演示稳定性。

### 图标和素材
- 用户提供的 icon、插图应先复制进项目资源目录，再通过相对路径引用；不要在演示文件中直接引用 `/Users/.../Downloads` 临时路径。
- 同一业务 icon 如区分线性/面性，要记录使用场景。例如个人中心左侧菜单可用面性图标，顶部入口和弹层列表按需求使用线性图标。
- 替换图标后需要检查 hover、active、白色反色、蓝色主色等状态是否仍能正常显示。

### 文案和按钮规范
- 两个汉字按钮文案按组件规范保留空格，例如 `导 出`、`取 消`、`确 定`。
- 展示型字段文案必须严格按用户给出的格式，不自行改写标点、括号或分隔符。
- placeholder 文案应按业务对象定制，不使用泛化占位文案。

### 验证规范
- 每次修改后至少运行脚本语法检查。
- 涉及页面 UI 或交互时，应使用浏览器打开本地 HTML 验证 DOM、路由、点击、hover、遮罩和关键尺寸。
- 涉及 Figma 组件尺寸时，应记录读取到的 Figma 节点名称和核心尺寸，并用浏览器实测本地渲染结果。

### 导出演示包
- 给别人演示时，应导出可直接打开的本地包。
- 给外部演示前，应先确认演示状态，例如消息是否需要全部未读、默认路由是否正确。
- 导出包应包含主 HTML 和实际引用到的资源文件，并保持原相对路径。
- 不要把无关研究截图、历史素材、临时文件一起打包。
- 导出后必须检查 zip 内容、文件大小，并对导出目录里的 HTML 做一次脚本语法检查。

## 禁止事项
- 不要删除已有功能。
- 不要随意改公共样式。
- 不要一次性修改大量无关文件。
- 不要引入复杂依赖，除非明确说明原因。
